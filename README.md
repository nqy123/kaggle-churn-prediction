# Kaggle 客户流失预测：从 baseline 到 XGBoost Optuna 多种子稳定方案

![Result Card](assets/result_card.png)

## 这个项目在简历里的定位

这是一个非常适合展示 tabular 建模全流程的项目：有真实业务目标、有类别/数值混合特征、有 OOF 验证、有 Optuna 调参、有多模型融合，也有线上提交后的取舍复盘。它体现的不只是“会跑模型”，而是能把一个业务预测问题完整拆解成数据理解、特征工程、模型验证和结果落地。

## 业务问题与评价指标

目标是根据客户属性预测是否流失，目标列为 `Churn`，评价指标为 ROC-AUC。业务上可以理解为客户运营问题：提前识别高流失风险客户，把优惠、客服、续约提醒等资源优先投向最值得挽留的人群。

当前最佳保留版本为 `outputs/submission_xgb_optuna50_multiseed.csv`，OOF AUC 为 `0.917168`。CatBoost baseline OOF AUC 为 `0.915442`，FE 版本为 `0.915505`，最终提升主要来自 XGBoost Optuna 与多 seed 稳定化。

## 我的整体解决思路

我把这个项目拆成五步：

1. **先理解数据**：看目标分布、字段类型、缺失模式、train/test 差异和指标特点。
2. **再做清洗**：把明显的脏值、哨兵值、语义缺失和泄露风险处理干净。
3. **重点做特征工程**：不是机械造列，而是把业务直觉、数据结构和模型能力连接起来。
4. **用 OOF 做验证**：每个模型都尽量通过交叉验证产生 OOF，避免只看一次线上提交。
5. **最后复盘取舍**：线上不涨的复杂方案不保留，最终只保留稳定、能解释、能复现的版本。

## EDA 与数据理解

- 先明确 `id` 是样本标识，`Churn` 是目标列，二者都不能进入训练特征。
- 数据同时包含类别特征和数值特征，因此需要两条建模路线：CatBoost 原生类别处理，以及 LightGBM/XGBoost 的编码版本。
- ROC-AUC 关注高风险客户排序能力，和客户挽留场景高度一致。
- FE 相比 baseline 只提升 `0.000063`，说明原始特征已经很强，后续提升更依赖调参和 seed 平均。

## 数据清洗策略

- 统一检查训练集、测试集和 sample submission 的行数、列名和顺序。
- 所有 submission 都检查预测范围、缺失值、无穷值，确保可以直接提交。
- 类别特征在 train/test 合并后统一编码，再拆分，避免编码错位。
- 严格防止数据泄漏：`id` 只用于输出，`Churn` 只用于目标和验证。

## 特征工程：不是堆字段，而是写入业务逻辑

| 特征设计 | 为什么这样构造 | 预期收益 |
| --- | --- | --- |
| 类别特征保留 | CatBoost 能直接学习类别统计模式，减少手工编码损失。 | 建立强 baseline，保留类别信息。 |
| 统一编码版本 | XGBoost/LightGBM 需要数值输入。 | 让三类树模型可以公平比较。 |
| 业务交叉与分箱 | 流失往往由状态、费用、服务使用等因素叠加触发。 | 增强高风险客户群识别能力。 |
| 多 seed 预测 | 单次 CV 划分存在随机波动。 | 降低方差，提高线上提交稳定性。 |

这部分是整个项目最值得讲的地方。我的处理原则是：**先解释变量为什么可能影响目标，再决定把它做成比例、计数、交叉、分箱、rank、文本向量还是序列语义表示**。这样做出来的特征不是“玄学调参”，而是有业务含义、有验证闭环的建模资产。

## 模型选择：为什么用这些模型

- CatBoost 作为类别特征友好的强 baseline，快速验证数据可学习性。
- LightGBM 与 XGBoost 作为互补树模型，比较不同切分、正则和采样策略。
- Optuna 用于集中搜索最有潜力的单模型，而不是手工盲调。
- 最终选择 XGBoost Optuna50 MultiSeed，因为它在 OOF 和线上反馈之间最稳定。

我没有把模型当成黑箱堆叠，而是根据数据形态选择模型：结构化数据优先树模型，类别特征多时重视 CatBoost，高维匿名数值用 LightGBM，文本任务则先用 TF-IDF 建强 baseline，再用 Transformer 补语义理解。

## 验证与防止过拟合

- 使用 OOF 或交叉验证观察本地泛化表现。
- 区分 public/private 分数，避免过度追逐单次 public leaderboard。
- 严格排除 ID、target、后竞赛标签等泄露来源。
- 对融合方案做线上复盘：如果复杂方案不涨分，就回退到更稳定版本。

## 运行结果分析

- FE 相比 baseline 提升有限，说明普通特征工程边际收益小。
- XGBoost Optuna50 MultiSeed 的 OOF AUC 达到 `0.917168`，是当前最稳泛化选择。
- 更复杂的 fine pseudo blend 线上没有继续涨分，因此最终回到可复现、风险更低的版本。

## 如何复现

安装依赖：

```bash
pip install -r requirements.txt
```

复现时先从 Kaggle 下载原始数据到项目约定的数据目录。部分仓库为了保持轻量，只保留最佳提交文件、实验日志和核心说明；如果仓库中存在 `src/`、`notebooks/` 或 `kaggle_kernel_*`，优先从这些入口运行训练。

常见入口示例：

```bash
python src/train_best.py
# 或在 Kaggle 上运行 kaggle_kernel_* 中的 GPU kernel
```

如果当前项目只保留了最佳产物，可直接查看 `outputs/` 中的 OOF、prediction、submission 和实验摘要文件。
